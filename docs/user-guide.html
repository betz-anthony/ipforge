IPForge User Guide

IPForge is a self-hosted IP address management tool with DNS and DHCP integration. It tracks subnets and IP addresses, syncs live data from your DNS/DHCP servers, and exposes an allocation API for Terraform and Ansible.

Deployment

Docker Compose (recommended)

cp .env.example .env
# Edit .env — see Environment Variables below
docker compose up -d

The app is available at http://localhost (nginx on port 80). API docs at /api/docs.

Environment Variables

Variable Default Description
DB_PASSWORD ipam PostgreSQL password
JWT_SECRET_KEY change-me-in-production-... Change this. Signs auth tokens.
SECRET_KEY (empty) Fernet key for encrypting provider credentials at rest. Generate: python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())". If empty, credentials stored as plaintext.
DEFAULT_ADMIN_PASSWORD admin Initial admin password. Change on first login.
JWT_EXPIRE_MINUTES 480 Token lifetime (8 hours).
SYNC_MODE background background = Docker Compose (built-in scheduler). disabled = Kubernetes (external CronJob handles sync).
SCAN_INTERVAL_MINUTES 30 Default ping-sweep interval for subnets with no per-subnet override.
UTIL_WARN_THRESHOLD 80 Utilization % that turns subnet bar yellow.
UTIL_CRITICAL_THRESHOLD 95 Utilization % that turns subnet bar red.

MS WinRM Variables (if using MS DNS / MS DHCP)

Variable Example
MS_WINRM_HOST dc01.corp.local
MS_WINRM_USER CORP\svcipam
MS_WINRM_PASSWORD (service account password)
MS_WINRM_TRANSPORT ntlm
MS_DNS_SERVER dns01.corp.local
MS_DHCP_SERVER dhcp01.corp.local

WinRM must be enabled on the target host (Enable-PSRemoting). The WinRM host can be a jumpbox — it doesn't need to be the DNS/DHCP server itself.

First Boot

  1. Navigate to the app and log in with admin / admin (or DEFAULT_ADMIN_PASSWORD if set).
  2. You will see a warning banner — change the admin password immediately via Settings → Users.
  3. Set JWT_SECRET_KEY and SECRET_KEY in .env and restart if not already done.
  4. Add at least one DNS and/or DHCP provider in Settings → Providers.
  5. Trigger an initial sync (Settings → Providers → Sync Now) to populate DNS/DHCP caches.

Roles

Five roles. read-only, operator, and admin are nested (each a superset of the one below). requester and scoped are orthogonal — granted alongside global limits.

Role Can do
requester Submit IP requests for subnets flagged request-eligible and see only their own requests. Cannot view other resources.
scoped Read-only on the full app, plus full read/write only on subnets they have been granted access to (see Per-Subnet Grants).
read-only View all pages, search, browse DNS/DHCP/audit data. Cannot write anything.
operator Everything read-only can do, plus: create/edit/delete subnets and addresses, trigger scans, allocate IPs, import CSVs, add/delete DNS records and DHCP reservations, approve/deny IP requests, acknowledge alerts.
admin Everything operator can do, plus: manage users + groups, manage provider configurations, manage alert channels + rules, change utilization/scan/stale thresholds, clear provider caches, mint API tokens, grant per-subnet access.

Manage users at Settings → Users (admin only).

Per-Subnet Grants

For finer-grained access, the scoped role pairs with per-subnet grants. Admins open a subnet → Access section → add a grant for a user or group with view or manage permission. A scoped user gets read-only access to the rest of the app and full write access only to the subnets they (or their groups) have manage on.

Manage groups at Settings → Groups — assign users to groups, then grant subnet access by group rather than per-user.

Provider Configuration

IPForge connects to external DNS and DHCP servers to read and write live data. Providers are configured in the UI — no env var changes needed after initial deployment.

Adding a provider

  1. Go to Settings → Providers.
  2. Click Add Provider.
  3. Select category (DNS or DHCP) and type (see below).
  4. Fill in connection details. Credentials are encrypted at rest if SECRET_KEY is set.
  5. Enable the provider and set sort order (lower number = higher priority for auto-selection).

Supported DNS providers

Type Notes
msdns Microsoft DNS via WinRM/PowerShell. Requires MS WinRM variables.
bind BIND 9 via AXFR (read) + RFC 2136 dynamic update (write).
pihole Pi-hole v5/v6. Read-only for zones; record creation supported.
cloudflare Cloudflare API v4. Auth: API token (Zone:DNS:Edit).
route53 AWS Route 53. Auth: access key + secret (optional region).
azure_dns Azure DNS. Auth: service principal (tenant/client/secret + subscription + resource group).
gcp_dns Google Cloud DNS. Auth: service-account JSON + project ID.

Supported DHCP providers

Type Notes
msdhcp Microsoft DHCP via WinRM/PowerShell. Requires MS WinRM variables.
keadhcp ISC Kea via Control Agent (dhcp4 + dhcp6; DUID reservations for IPv6).
pihole Pi-hole DHCP.

Sync

IPForge caches DNS zones/records and DHCP scopes/leases locally for fast lookups. The cache is refreshed on a background schedule.

  • Manual sync: Settings → Providers → Sync Now (or per-category).
  • Sync status shows last sync time and any errors on the Dashboard.
  • Clear cache: Settings → Providers → Clear Cache button (per provider, admin only). Use this when removing a provider or after major DNS/DHCP changes that haven't propagated.

Subnets

Viewing subnets

The Subnets page has two view modes:

  • Flat table — all subnets listed with utilization bars. Default view.
  • Tree view — hierarchical two-panel layout. Left panel shows the subnet tree; right panel shows addresses in the selected subnet. Toggle with the Tree/Flat button.

Each subnet row shows: - CIDR and name - Utilization bar (% of IPs with non-available status). Yellow at warn threshold, red at critical threshold. - Rollup utilization (includes child subnets) - Parent badge (links to parent subnet) - Collision badge (count of unresolved collisions — click to resolve)

Click any column header to sort (CIDR sorts numerically across IPv4 and IPv6). Use the search box above the table to filter by name, CIDR, VLAN, or description.

Creating a subnet

Click Add Subnet. Fields:

Field Notes
Name Display name
CIDR e.g. 10.0.0.0/24. Parent subnet auto-suggested.
VLAN ID Optional
Description Optional
Notes Optional free text
Scan Interval Minutes between automatic ping sweeps. Overrides the global setting. Leave blank to use global.
DNS Provider Optional. Name of the provider to use for allocation DNS registration in this subnet. Overrides the first-enabled provider.
DHCP Provider Optional. Name of the provider to use for allocation DHCP registration in this subnet.
Parent Subnet Auto-suggested from CIDR. Manually selectable.

Subnet hierarchy

Subnets can be nested. A /16 can contain multiple /24s. Creating a /24 when a /16 already exists will auto-suggest the /16 as the parent.

Rules: - Parent must strictly contain the child CIDR. - Cycles are rejected. - Deleting a subnet that has children is rejected (409) — delete children first.

Editing a subnet

Click any subnet row to open the detail drawer. Edit fields inline and click Save.

Scanning a subnet

From the subnet detail drawer or the Subnets page, click Scan. This triggers an ICMP ping sweep of all host IPs in the subnet. Results update last_seen on each address and feed the 7-day availability history.

Large subnets: the UI warns if the subnet is too large for a quick scan (/24 max for IPv4, /120 for IPv6). You can still scan — it just takes longer.

Reserved Ranges & Subnet Map

Each subnet's detail drawer has a Reserved Ranges editor and an Address Map.

Reserved ranges

Define named, typed ranges inside a subnet so allocation skips them and operators can see committed space:

Kind Typical use
gateway router / first-hop addresses
dhcp_pool the dynamic DHCP pool
static a block held for manual static assignment
reserved anything else held out of allocation

Add a range with a start IP, optional end IP (defaults to start = a single IP), kind, and label. The Allocation API never hands out an IP inside a reserved range. Reserved IPs are counted as a separate bucket in the subnet's utilization (shown as a hatched segment on the utilization bar), distinct from used. Writing ranges requires operator+. Validation rejects ranges outside the subnet CIDR, reversed start/end, or overlaps with an existing range.

Address map

A grid of every host IP in the subnet, colored by status (free / assigned / reserved / discovered / deprecated) with a red outline on any IP that has open drift. Click a cell to inspect it; a free cell offers Create address here. Subnets larger than 1024 hosts show a "drill into a child subnet" message instead of a giant grid.

Capacity Forecasting

IPForge records a daily snapshot of each subnet's used/total counts and projects when the subnet will fill.

  • Per-subnet — the subnet drawer shows "exhaustion in ~N days" with a projected date and a confidence level, computed by least-squares regression over the daily history.
  • Dashboard — a Projected Exhaustion widget lists the subnets with the soonest projected critical-threshold breach.
  • API — GET /api/subnets/{id}/forecast returns slope (IPs/day), current/total, days-to-warn, days-to-critical, projected dates, and confidence (none/low/medium/high). GET /api/subnets/forecasts?limit=N returns the top-N soonest-exhaustion subnets.

Confidence is none when usage is flat or declining (no exhaustion projected) or when there is not yet enough history (snapshots accrue once per day).

VLANs

The VLANs page is a lightweight catalog of 802.1Q VLAN IDs with human-readable names. It's decoupled from the subnet table — a subnet's vlan_id is still a plain integer, but when a VLAN entry exists for that ID the Subnets page shows it as 100 (Production) instead of just 100.

Creating a VLAN

Click Add VLAN. Fields:

Field Notes
VLAN ID 1–4094 (12-bit, 0 and 4095 reserved). Must be unique.
Name Human-readable label. Required.
Description Optional one-liner.
Notes Optional free text.

Editing / deleting

Click the pencil icon to edit, trash to delete. Delete is disabled while any subnet still carries that VLAN ID — change those subnets first.

Counts

The Subnets column shows how many subnets currently reference each VLAN ID. Sortable so heavily-used VLANs surface to the top.

Permissions

  • Any authenticated user can list and read VLANs
  • Operator+ can create, edit, delete

IP Addresses

Viewing addresses

Go to Addresses or open a subnet and click into the tree view. Filter by subnet or status using the controls at the top. Sort by any column header (IPs sort numerically) and use the search box to filter by IP, hostname, MAC, or description.

Address statuses:

Status Meaning
available IP is free (no active use recorded)
reserved Manually reserved or allocated via the API
assigned In active use
deprecated Retired — not available for allocation
discovered Found by a scan but not otherwise registered

Address drawer

Click any address row to open the drawer. Shows: - IP, hostname, MAC, status, description, notes - last_seen timestamp (last successful ping) - 7-day reachability dot row (filled = reachable that day) - DNS records matching this IP (from cache) - DHCP reservations matching this IP (from cache) - Network location — discovered switchport / VLAN / device, if SNMP Discovery is configured - Custom field values and tags (Custom Fields & Tags) - History — a merged lifecycle timeline (changes from the audit log, drift events, reachability up/down) plus an as-of date picker that reconstructs what the address looked like on any past date ("on 2026-01-01: assigned to web-01"). Works even for IPs no longer tracked, via GET /api/addresses/by-ip/{ip}/history.

Creating an address

Click Add Address. Manually specify the IP and status. For automated allocation use the Allocation API instead.

Collision detection

The three classic collision types (active_but_available, hostname_mismatch, multi_dhcp_scope) are now part of the broader Drift / Reconciliation engine, alongside DNS/DHCP reconciliation categories. Collisions still appear as badges on subnet rows and a Dashboard card with the Guided Resolve dialog; the full picture lives on the Drift page.

Custom Fields & Tags

Attach your own metadata to subnets and addresses.

Custom fields (admin-defined)

In Settings → Custom Fields, an admin defines fields per entity type (subnet or address):

  • name — machine key (lowercase, digits, underscore), unique per entity type.
  • label — display name.
  • type — text, select (with a fixed option list), or date.

Defined fields then appear in the subnet/address drawers as inputs. select values are validated against the option list; an empty value clears the field. Deleting a field definition removes all its stored values.

Tags

Free-form tags (many-to-many) on subnets and addresses — type comma-separated tags in the drawer; new tags are created automatically. Creating tags requires operator+.

Filtering & export

List endpoints support server-side filtering: GET /api/subnets?tag=prod&cf_env=production (and the same on /api/addresses). CSV export adds a tags column plus a cf_<name> column per defined field.

IP Requests

Non-operators (the requester role) can ask for an IP without touching the allocation API directly. Operators review and approve from a queue.

Submitting (requester)

  1. Open the Requests page → Request IP.
  2. Select a request-eligible subnet from the dropdown. Admins flag eligible subnets with the Request-eligible checkbox on the subnet edit form — only those appear here.
  3. Fill in hostname, purpose, and (optionally) MAC. Submit.
  4. The request shows up in your list with status pending. You can cancel pending requests.

Reviewing (operator / admin)

  1. Open the Requests page — default filter is pending.
  2. Approve — allocates the next free IP in the selected subnet (uses the Allocation API internally). Tickboxes during approval let you also register DNS, DHCP, and PTR in one step.
  3. Deny — requires a reason; written to notes and notified to the requester (if alerting configured).

Both actions write to the audit log and fire ip_request_resolved alerts; submission fires ip_request_submitted.

Allocation API

The allocation API lets automation tools (Terraform, Ansible, scripts) claim the next free IP in a subnet without manual intervention.

Endpoint

POST /api/subnets/{subnet_id}/allocate

Requires operator+ token.

Request body

{
  "hostname":      "web-01",
  "description":   "",
  "mac_address":   "aa:bb:cc:dd:ee:ff",
  "notes":         "",
  "register_dns":  false,
  "register_dhcp": false,
  "dns_zone":      "example.com",
  "dns_provider":  "",
  "dhcp_provider": ""
}

hostname is required. All other fields are optional.

How IP selection works

  1. Loads all ineligible IPs in the subnet (reserved, assigned, deprecated, discovered).
  2. Walks ipaddress.hosts() ascending — this excludes the network address and broadcast address.
  3. Skips .1 (router convention) and .255 (broadcast-like convention).
  4. Returns the first IP not in the ineligible set.

Idempotency

Hostname is the idempotency key (case-insensitive). If web-01 is already allocated in this subnet with a non-deprecated status, the existing record is returned — no second IP is created. This makes the endpoint safe to re-run from Terraform or Ansible.

  • First call → 201 Created, is_new: true
  • Subsequent calls with same hostname → 200 OK, is_new: false

If mac_address is provided on a repeat call and the existing record has no MAC, the MAC is updated.

DDI registration

Set register_dns: true and/or register_dhcp: true to register the IP at allocation time.

Flag Required fields Action
register_dns: true dns_zone Creates an A record: hostname → IP in the specified zone
register_dhcp: true mac_address Creates a reservation in the DHCP scope that contains the allocated IP

Rollback on failure (new allocations only): If DNS registration fails, the IP is not committed. If DHCP registration fails, the DNS record is deleted and the IP is not committed. This keeps IPAM, DNS, and DHCP consistent.

Idempotent hits: If the hostname already exists, DNS/DHCP registration still runs (re-registration after partial failure), but failure does not affect the existing IP record.

Provider selection

For both dns_provider and dhcp_provider:

  1. Explicit value in request body
  2. dns_provider_name / dhcp_provider_name on the subnet record (set this per-subnet in the subnet form)
  3. First enabled provider of that category

Response

{
  "id":              42,
  "address":         "10.0.1.5",
  "subnet_id":       1,
  "subnet_cidr":     "10.0.0.0/16",
  "hostname":        "web-01",
  "status":          "reserved",
  "mac_address":     "aa:bb:cc:dd:ee:ff",
  "dns_registered":  true,
  "dhcp_registered": false,
  "is_new":          true
}

Getting a token

curl -s -X POST https://ipam.example.com/api/auth/login \
  -d "username=svcaccount&password=yourpassword" \
  | jq -r .access_token

Terraform example

See docs/examples/terraform/main.tf. Uses the Mastercard/restapi provider. Key points: - read_path = "/addresses/{id}" — Terraform tracks the resource by database ID, so re-apply after state loss does a GET /addresses/{id} rather than re-allocating. - destroy_path = "/addresses/{id}" — terraform destroy deletes the address record (sets it back to available). - id_attribute = "id" — tells the provider which field carries the stable ID.

Ansible example

See docs/examples/ansible/allocate.yml. Key point: - status_code: [200, 201] — the task succeeds on both new allocation and idempotent hit. Playbooks are safe to re-run.

DNS

The DNS page shows zones and records synced from your configured DNS providers.

Zones

Zones are listed on the left. Each zone is tagged with its source provider. Click a zone to load its records.

Zones are grouped into three tabs: Forward, Reverse, and Trust (PTR/stub zones).

Records

Records are displayed with name, type, value, TTL, and source. Use the search box to filter within a zone.

Adding a record

Click Add Record. Specify name, type (A, AAAA, CNAME, MX, etc.), value, and TTL. The record is written to the provider immediately.

Deleting a record

Click the delete icon on any record row. Confirmation required.

Note: DNS data shown is from the local cache. After adding or deleting records, the cache updates on the next sync cycle. Use Sync Now to force an immediate refresh.

PTR records

When you add an A or AAAA record, the Also create PTR checkbox writes the matching reverse-zone record in one step. Deleting an A/AAAA prompts you to delete its PTR via the delete preview dialog (preview lists every provider record that will be removed; uncheck any you want to keep).

PTR is also auto-created when the Allocation API is called with register_ptr: true, or when an operator approves an IP request with the PTR checkbox ticked.

The reverse zone must already exist on the DNS provider. PTR creation is silently skipped if no reverse zone covers the IP.

DHCP

The DHCP page shows scopes and leases/reservations synced from your DHCP providers.

Scopes

Scopes are listed with subnet mask, range, active status, and source provider.

Leases / Reservations

Click a scope to see its leases and reservations. Shows IP, MAC, name, and description.

Adding a reservation

Click Add Reservation from within a scope. Specify IP, MAC, name, and description. The reservation is written to the DHCP provider immediately.

Deleting a reservation

Click the delete icon on any reservation row. Confirmation required.

Subnet Scanning

IPForge can ping-sweep subnets to discover active hosts.

On-demand scan

From the Subnets page, open a subnet and click Scan. A real-time status indicator appears. Results are available when the scan completes.

Continuous scanning

Each subnet can have an automatic scan interval. Set Scan Interval on the subnet (in minutes). Leave blank to use the global default (Settings → Scan Interval). Set to 0 to disable scanning for that subnet.

The background scheduler fires every 60 seconds and runs pending scans.

What scanning does

  • Sends ICMP pings to each host IP.
  • Updates last_seen on address records that respond.
  • Creates discovered records for IPs that respond but aren't in the database.
  • Records daily availability aggregates (used by the 7-day dot row in the address drawer).
  • Fires scan alerts when a host transitions: went_unreachable or came_back.

Scan alerts

Alerts appear on the Dashboard. Each alert shows the IP, event type, and timestamp. Acknowledge individually or bulk-acknowledge all (with optional subnet filter).

For routed notifications (email/Slack/Teams/PagerDuty), configure an alert channel + rule — see Alerting.

Drift / Reconciliation

Drift is the gap between IPAM's intended state and the actual state seen in DNS, DHCP, and live scans. The Drift page surfaces all discrepancies in one place and lets you remediate them.

Categories

Category Severity Meaning
active_but_available warning IP responds to ping but is marked available
hostname_mismatch warning IPAM hostname differs from DHCP/DNS
multi_dhcp_scope warning same IP reserved in multiple DHCP scopes
missing_dns warning assigned/reserved IPAM address has no DNS A/AAAA record
orphan_dns info DNS A/AAAA record with no IPAM address
orphan_dhcp info DHCP lease/reservation with no IPAM address
mac_mismatch warning IPAM MAC differs from the DHCP reservation MAC

Detection

Drift is recomputed automatically at the end of each subnet scan, after every DNS/DHCP sync, on a periodic background pass, and on demand via Scan for drift (or POST /api/drift/scan). An item that no longer holds is auto-resolved.

Remediation

Each row offers the right action for its category (provider changes roll back on failure):

  • conflict types → Guided Resolve (status update / push canonical hostname / remove duplicate reservation).
  • orphan_dns / orphan_dhcp → Import (create the IPAM address) or Delete (remove the record/reservation).
  • mac_mismatch → Use DHCP MAC (copy into IPAM).
  • any → Dismiss.

Select multiple rows of the same kind for bulk dismiss. Filter by category; the stats header shows counts by category and severity.

Auto-remediation (Settings → Drift Auto-Remediation)

IPForge can fix open drift automatically, per category, with conservative guardrails. In Settings → Drift Auto-Remediation (admin) set each category's mode:

  • auto (safe, IPAM-only categories only — orphan_dns / orphan_dhcp → import, mac_mismatch → copy DHCP MAC, active_but_available → set a status): the action is applied automatically each sweep.
  • review: matching items are flagged needs review (a badge + filter on the Drift page) for a human — used for destructive / provider-writing categories (deletes, hostname push, missing-DNS create), which are never automated.
  • off: no policy.

Guardrails: new auto policies default to dry-run (they log what they would do until you turn it off); every action is audited; actions are reversible and IPAM-only; and gitops-managed resources are skipped — the declared document is authoritative for those. The sweep runs right after each drift detection pass.

API

GET /api/drift (filters: category, severity, subnet_id, resolved, needs_review), GET /api/drift/stats, POST /api/drift/scan, POST /api/drift/{id}/resolve, POST /api/drift/resolve-bulk, GET/PUT/DELETE /api/drift/policies[/{category}].

SNMP Discovery

Poll network devices over SNMP to learn where each host actually lives — IP ↔ MAC ↔ switchport ↔ VLAN — and enrich IPAM addresses with that ground truth.

Configuring devices

In Settings → Discovery Devices (SNMP) (admin), add a device:

  • host, SNMP version (v2c or v3).
  • v2c: community string.
  • v3: user, optional auth (SHA/MD5 + key) and priv (AES/DES + key).
  • poll interval (minutes).

Credentials are stored encrypted (when SECRET_KEY is set) and never returned by the API. Devices are polled on their interval, or on demand with Poll. Each device shows its last run status.

What it collects (P1)

Per poll, IPForge walks standard MIBs: ARP (ipNetToMediaPhysAddress), the bridge forwarding table with VLAN (dot1qTpFdbPort), bridge-port→ifIndex, and ifName. It merges these into endpoints (IP, MAC, switchport name, VLAN).

Enrich-only

Discovery does not create IPAM addresses. For addresses already tracked, the detail drawer's Network location block shows the discovered switchport, VLAN, and device (matched by IP or MAC). Unknown IPs/MACs surface through Drift instead. GET /api/discovery/endpoints lists raw discovered endpoints.

Roadmap: SSH/CLI collection, CDP/LLDP topology, and drift categories for mac-on-unexpected-port / rogue devices are planned follow-ups.

Security Events

The Security page surfaces network-security signals derived from scans and SNMP discovery:

Event Meaning
rogue_device A device (IP/MAC) seen on the network with no IPAM record.
mac_move A MAC moved to a different switchport/device since the last SNMP poll.
ip_conflict One IP claimed by two or more different MACs (possible spoof).
new_mac A previously-unseen MAC appeared.

Detection runs after each SNMP poll (and scan rogue detection); duplicate events are de-duplicated rather than re-created. Each event has a severity (error/warning/info).

Acknowledge clears an event from the default view. Quarantine (operator+) ensures an IPAM address exists for the offending IP, tags it quarantined, and sets its status to deprecated — reversible and audited. (SNMP port-shutdown enforcement is a future addition.)

GitOps (Declarative Config)

Declare VLANs, subnets, reserved ranges, and allocations as YAML and have IPForge reconcile its state to match. One-way (declared → IPForge); your CI/CD posts the document. IPForge stores no git credentials — git history is the record of truth.

Document

source: prod-net            # required; namespaces what this doc manages
vlans:
  - { vlan_id: 10, name: servers }
subnets:
  - cidr: 10.0.0.0/24
    name: servers
    vlan_id: 10
    parent: 10.0.0.0/16      # optional, by CIDR
    reserved_ranges:
      - { start: 10.0.0.1, kind: gateway, label: gw }
allocations:
  - { subnet: 10.0.0.0/24, hostname: web-01, register_dns: false }

Plan & apply

  • Plan (POST /api/gitops/plan) — dry-run; returns a per-resource diff of what would be created / updated / pruned. No changes made.
  • Apply (POST /api/gitops/apply) — reconciles: VLANs, then subnets (parents first), reserved ranges, then allocations (via the Allocation API, honoring register_dns/dhcp).

Both accept the YAML document as the request body (Content-Type: text/yaml; JSON also works) and require operator+. The GitOps page in the UI provides a YAML editor with Plan and Apply buttons.

curl -X POST https://ipforge.example.com/api/gitops/apply \
  -H "Authorization: Bearer ipfg_<token>" \
  -H "Content-Type: text/yaml" \
  --data-binary @network.yaml

Managed resources & prune

Every resource IPForge creates or updates from a document is tagged with that document's source. On apply, resources of the same source that have disappeared from the document are pruned. Resources created manually, discovered/synced from providers, or owned by a different source are never touched. Re-applying an unchanged document is a no-op (idempotent).

Providers (DNS/DHCP) are never pruned by GitOps — if an allocation is removed from the document, at most the IPForge IPAM row is pruned; any DNS/DHCP record left behind surfaces as drift, not a silent delete.

Stale IP Reclamation

Addresses that haven't responded to a scan for a configurable number of days are flagged stale. The Dashboard shows a count; the Reclaim page lists them grouped by subnet.

How staleness is decided

  • Threshold set globally in Settings → Stale Reclaim Threshold (days). 0 disables the feature.
  • An address is stale when last_seen is older than threshold AND its status is assigned or reserved (free / deprecated / discovered are not candidates).
  • The check runs as part of every scan; counts are cached.

Reclaim workflow

From the Reclaim page, for each stale IP: - Mark deprecated — sets status to deprecated so it won't be re-allocated but stays in the database for audit. - Delete with cleanup — opens the delete-preview dialog and removes the IPAM record plus any DNS/DHCP records you tick. - Ignore — clears the stale flag without changing status (resets the timer for one cycle).

When the stale queue depth crosses configured rule conditions, a stale_queue alert fires.

Rules (Settings → Alert Rules)

A rule binds one or more channels to a trigger with optional condition + re-notify cadence.

Trigger Fires when
collision Any new unresolved collision is detected
utilization A subnet's utilization crosses the rule threshold (%)
rogue A scan finds a responsive IP that doesn't match any expected source
sync_error A DNS/DHCP provider sync fails
stale_queue Stale-IP queue depth exceeds the rule threshold
ip_request_submitted A requester submits a new IP request
ip_request_resolved A pending request is approved or denied

Re-notify — minutes between re-sends while an event remains in firing state. Empty = notify once per state change.

Events (Alerts page)

Lists every alert event with deliveries (✅ sent / ❌ failed per channel). Filter by state (firing / resolved) and trigger. Operators can acknowledge a firing event to stop re-notifications.

Automation rules (Settings → Automation Rules)

Beyond notifying, IPForge can act on a trigger. In Settings → Automation Rules (admin) define rules that, on a rogue (newly-discovered device) or drift event, tag and/or set the status of the matching address — e.g. tag every newly-discovered device unverified, or set orphan_dhcp drift IPs to reserved. Drift rules can be scoped to a single drift category.

Actions are reversible and IPAM-only (no provider writes, no deletes), each is audit-logged, and rules have a per-rule enable toggle. Rules run from the same background dispatcher that delivers alerts.

Search

The global search bar (top of every page) searches across subnets, addresses, DNS records, and DHCP leases simultaneously.

Results are grouped by type. Click any result to navigate to it.

Search is server-side with a 50-item cap per type. Results are debounced 250ms.

Import / Export

Bulk-load data from CSV or extract the full database for backup or migration.

Export

On the Subnets or Addresses page, click Export CSV. Downloads all records immediately.

Subnets CSV columns: name, cidr, ip_version, vlan_id, description, notes, parent_cidr, scan_interval_minutes, dns_provider_name, dhcp_provider_name

Addresses CSV columns: address, subnet_cidr, hostname, status, mac_address, description, notes

Import

Click Import CSV on the Subnets or Addresses page. Select your file. The import upserts — existing records are updated, new records are created.

Key rules: - Subnets are keyed by CIDR. Addresses are keyed by IP address. - parent_cidr is resolved at import time — if the parent exists in the database or appears earlier in the same file, the relationship is set automatically. - Address status defaults to available if omitted. - Rows with validation errors are skipped; a per-row error list is shown in the result banner. - Import requires operator role.

Import order matters for hierarchical subnets: parents must appear before children in the CSV, or already exist in the database.

LDAP / Active Directory Authentication

IPForge supports optional LDAP/AD authentication alongside local accounts.

How it works: - Users with a local password always authenticate locally (LDAP is not consulted). - Users without a local password are authenticated against the configured LDAP server. - On first successful LDAP login, a shadow account is created in IPForge with no local password. - Role is refreshed from LDAP group membership on every login.

Setup (Settings → LDAP / Active Directory):

Field Description
LDAP Host Hostname or IP of your domain controller
Port 389 (LDAP) or 636 (LDAPS)
Use SSL Enable for LDAPS
Bind DN Service account DN: cn=svc-ipam,dc=example,dc=com
Bind Password Service account password (stored encrypted if SECRET_KEY env var is set)
Base DN Search root: dc=example,dc=com
User Filter LDAP filter with {username} placeholder. Default: (sAMAccountName={username})
Admin/Operator/Readonly Group DN Optional — group membership determines role
Default Role Role assigned when no group DN matches

Role mapping: Groups are checked in order: Admin → Operator → Readonly. The first match wins. If no groups are configured, all LDAP users get the Default Role.

Local accounts are always preserved. The admin account created at first startup is a local account and is unaffected by LDAP configuration.

MCP Server (AI agents)

IPForge ships an MCP (Model Context Protocol) server so AI agents — Cursor, Continue, or any MCP-compatible client — can operate IPAM directly: "find a free IP in 10.0.0.0/24", "allocate web-07 in the DB subnet", "what's drifting in DC1".

It runs as a separate stdio process that talks to IPForge over the HTTP API with an API token. The token's role is the authorization boundary — a read-only token can list/search but the write tools return 403.

Install + run (in its own environment — it pins a newer dependency than the web app, so keep it separate):

pip install -r backend/requirements-mcp.txt
IPFORGE_URL=https://ipforge.example.com IPFORGE_TOKEN=ipfg_xxx \
  python -m app.mcp_server

MCP client config (most stdio-MCP clients accept this shape):

{
  "mcpServers": {
    "ipforge": {
      "command": "python",
      "args": ["-m", "app.mcp_server"],
      "env": { "IPFORGE_URL": "https://ipforge.example.com", "IPFORGE_TOKEN": "ipfg_xxx" }
    }
  }
}

Tools: list_subnets, get_subnet, list_addresses, find_free_ip, search, list_drift, list_discovery, ip_history (read); allocate_ip, tag_address, resolve_drift (writes — require an operator+ token).
